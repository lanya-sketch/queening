# 검증 하네스

실제 브라우저로 앱을 구동해 회귀를 확인하는 스크립트 모음.
설치된 Chrome/Edge 를 그대로 쓰므로 브라우저 다운로드가 없다(`playwright-core`).

**모든 스크립트는 개발 서버가 떠 있어야 한다.** 다른 터미널에서 먼저:

```bash
npm run dev
```

**돌아가는 동안 파일을 편집하지 말 것.** Vite HMR 이 페이지를 리로드하면 진행 중이던
빌드가 초기 상태로 돌아가 `최종 자원: {}` 같은 가짜 실패가 나온다. 특히 `simulate`(~8분)에서
FAIL 이 뜨면 먼저 그 빌드만 단독으로 다시 돌려보고(`npm run simulate -- A`) 판단하라.

## 환경변수

- `QUEENING_URL` — 개발 서버 주소 (기본 `http://localhost:5173/`)
- `QUEENING_CHROME` — Chrome/Edge 실행 파일 경로 (자동 탐색 실패 시)
- `QUEENING_LIVE_KEY` — **실제 키가 필요한 실측 두 가지에 쓴다.**

  ```
  QUEENING_LIVE_KEY=sk-ant-... npm run verify:chars            # 다섯의 목소리가 실제로 다른가
  QUEENING_LIVE_KEY=sk-ant-... npm run verify:live-affection   # 호감도가 회당 얼마나 오르는가
  ```

  `verify:live-affection` 은 회당 호감도 상승을 실측해 **깊은 관계(70)까지 몇 회가
  필요한지**를 계산하고, 세션 호출 상한(60)을 넘으면 조정 후보를 제시한다.
  키 없이 돌리면 무엇을 재는지 안내하고 종료한다(과금 없음).
  옵션: `QUEENING_LIVE_CHAR`(기본 prince) / `QUEENING_LIVE_ROUNDS`(기본 12) / `QUEENING_LIVE_MODEL`

  ★ 두 스크립트의 **배선은 네트워크를 가로채 검증했지만, 모델의 실제 행동은
  키 없이 확인할 수 없다.** 그게 이 스크립트들이 존재하는 이유다.

## 정적 대조 vs 진짜 ablation

둘은 다른 것을 증명한다. 섞어서 말하지 말 것.

| | 보는 것 | 놓치는 것 |
| --- | --- | --- |
| 정적 대조 (`verify:bloodoath` A절 등) | **의도** — 새 콘텐츠가 기존 flag 를 직접 참조하지 않음 | 우선순위 경쟁, 턴 예산(`MAX_EVENTS_PER_TURN=2`) 잠식 같은 **간접 영향** |
| ablation (`verify:ablation`) | **결과** — 실제로 들어내도 로그가 같음 | (느리다. 40분) |

**ablation 에는 결정론 모드가 필수다.** 활동 효과의 ±variance 때문에 같은 코드로 두 번
돌려도 로그가 갈리고, 그 상태로 비교하면 차이가 제거 탓인지 운 탓인지 못 가린다.
`QUEENING_DETERMINISTIC=1` 이 rng 를 상수 0.5 로 고정하는데, 이때
`floor(0.5 × (2v+1)) − v = 0` 이 모든 v 에서 성립해 **편차가 정확히 0** 이 된다.
스크립트 C 절이 매번 자기대조로 이 전제를 확인한다 — **C1 이 깨지면 B1 의 "일치"는 무의미하다.**

판정 시 주의: 제거한 콘텐츠에 자원 보상이 있으면(두루마리 +18) 그 보상을 받는 빌드는
**당연히 달라진다.** 그건 훼손이 아니라 ablation 이 작동했다는 증거다. 그래서
미스터리 결론(B절)과 자원 추이(D절)를 나눠서 판정한다.

디버깅용 환경변수:
- `QUEENING_DETERMINISTIC=1` — variance 0 (simulate 에 직접 걸 수도 있다)
- `QUEENING_ABLATE=bloodoath,devices,topics,incidents` — 그 콘텐츠를 런타임에서 제거
- `QUEENING_ABLATION_REPLAY=1` — 저장된 로그로 판정만 다시 (시뮬 재실행 없음)
- `QUEENING_ABLATION_MODE=incidents` — 돌발 현안 ablation (양쪽 팔 모두 돌발을 켠다)
- `QUEENING_ABLATION_MODE=hardexclusive` — 청산 ablation (결정적 씬은 시뮬에서 안 뜸)
- `QUEENING_INCIDENTS=1` — 시뮬에서 돌발을 켠다(고정 응답 가로채기 + 확률 0.9)

**돌발 ablation 은 확률을 정상보다 10배 이상 올려서 돌린다.** 정상 빈도(6~8%)는
결정론 모드(rng 0.5)에서 절대 통과하지 않아 "제거해도 같다"가 공허해진다.
과다 투여 상태에서 비교하는 편이 더 센 조건이기도 하다.

## ★ 실제 AI 호출을 내지 않는다

네트워크를 쓰지 않는 검증인데 **키를 설정하는** 스크립트가 여럿 있다. 그런 곳에서
AI 기능이 하나라도 자동 실행되면 진짜 호출이 나가고, 실제 키가 설정된 환경이면
**과금까지 된다.** 실제로 돌발 현안을 넣은 직후 `verify:devices` 가 401 을 내면서
이 구멍이 드러났다(그 테스트는 가짜 키를 쓰고 네트워크를 스텁하지 않았다).

그래서 키를 설정하는 스크립트는 둘 중 하나를 해야 한다:
- `page.route(...)` 로 응답을 스텁하거나
- `blockAiNetwork(page)` 로 원천 차단하고, 끝에서 차단 건수가 0 인지 단언하거나

돌발이 관심사가 아닌 검증은 `window.__queeningAi.setIncidentRate(0)` 으로 눌러 둘 것.
라이브 스크립트(`verify:live-affection`)도 재려는 것과 무관한 호출이 과금되지 않도록
같은 처리를 해 두었다.

## 콘텐츠를 추가한 뒤에 볼 것

`npm run simulate` 결과에서 **질적 결론**이 유지되는지가 핵심이다:

- A 빌드 = 깊은 진실 도달, B = 얕은 진실까지(재정 경로), C = 깊은 진실 차단
- D = 회유 성사(동맹), E = 영향도 0까지 떨어져도 회복 가능(함정 없음), F = 상한 근처 도달
- G = D 와 정책이 같고 **의심을 사냥 대회로만 낮추는** 대조군. D 와 나란히 놓고
  새 활동이 회유의 지름길이 되지 않았는지 본다(신망·영향도 차이를 볼 것)
- H = 침실 수색 성공 + 가문 강탈로 **혈서 확증(blood_oath_complete)** 까지 가는 루트
- I = ④ 호감도를 시드해 **두루마리(+18)** 가 엔드게임에 얼마를 보태는지 F 와 대조
- J = 명분 없이 처분을 강행하는 **폭군 루트** (H 는 명분을 들어 심판 = 정당 루트)
- K = ⑤ 결정적 씬 거절 → **측실** 선택 → 후일담 씬 실제 렌더(정의 검증 ≠ 렌더 검증 교훈)
  · ③ 정복 씬 렌더는 강빌드(영향도 60+)에서만 — 측실(영향도<60)과 정반대라 별도 확인

**모든 빌드는 끝에서 `judgeEnding()` 결과를 함께 찍는다.** "엔딩 없음"이 하나라도
나오면 `엔딩: *** 판정 실패 ***` 로 그 자리에서 드러난다. 네 처분 경로(정당·폭군·회유·못함)는
각각 H·J·D/G·나머지가 실제 플레이로 덮는다.

**플래너는 대화를 시뮬레이션하지 않는다.** 그래서 호감도로 열리는 콘텐츠(고유장치·혈서
로맨스 경로)는 기본 빌드에 **절대 나타나지 않는다** — 그게 정상이고, "로맨스 안 타면
이벤트 없음"의 실증이기도 하다. 그런 콘텐츠를 재려면 `seedAffection` 으로 시작값을
넣는 전용 빌드를 쓴다(I 빌드). 이건 로맨스 완주의 **대역**이지 대화 시뮬이 아니다.

**E 빌드의 "동맹"은 재현되지 않는다 — 그게 정상이다.** 담판 조건이 `의심 ≤ 45` 인데
E 는 의심 44 에서 담판에 들어간다. **1점 마진**이라 활동 variance 한 번이면 결렬로 뒤집힌다.
E 에 걸린 단언은 회유가 아니라 **영향도 0 → 회복**이고, 그건 어느 쪽으로 뒤집혀도 통과한다.
E 의 섭정 관계가 흔들린다고 회귀로 읽지 말 것.

**플래너의 기본값은 "첫 선택지"다.** 되돌릴 수 없는 선택지를 가진 이벤트를 추가하면
지정하지 않는 한 모든 빌드가 자동으로 그걸 고른다 — 실제로 「가문 수색」을 넣었을 때
B 빌드가 중립에서 결렬로 뒤집혔다. 그런 이벤트는 COMMON_CHOICES 에 보수적인 쪽을
등록하고, 파괴적인 쪽은 전용 빌드로 덮을 것.

**플래너가 안 쓰는 활동은 시뮬로 검증되지 않는다.** `planTurn` 은 정해진 정책만
따르므로, 활동을 추가했으면 그 활동을 실제로 고르는 빌드를 같이 넣어야 한다.
G 빌드가 그래서 있다 — 사냥 대회를 추가한 라운드에 6빌드를 그냥 돌렸다면
"밸런스 회귀 없음"은 아무것도 검증하지 않은 문장이었을 것이다.

**최종 수치가 1~2 흔들리는 것은 정상이다.** 활동 효과에 `variance`(±편차) 난수가 있고
시뮬 플래너가 매 턴 스탯을 읽어 적응하므로, 같은 코드로 두 번 돌려도 값이 미세하게 갈린다.
특히 임계값 기반 정책을 쓰는 D 빌드가 잘 흔들린다. 그러니 **"수치 회귀 0"을 기준으로 삼지 말고**,
질적 결론이 같은지 + 이벤트 조건 임계값이 설계와 일치하는지를 보라.

새 이벤트를 추가했다면 수치는 당연히 그 기여분만큼 달라진다. 그 차이가 새 이벤트로
설명되는지 확인하면 된다.
